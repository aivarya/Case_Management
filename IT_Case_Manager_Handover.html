1 Project Overview

The IT Case Manager is a full-stack web application built for internal IT departments to systematically manage support cases and tasks. It is designed to replace informal tracking methods (email, spreadsheets) with a structured, role-based ticketing system similar to Jira or Monday.com.

What It Does

  • Create, assign, and track IT support cases (tickets) with priority levels and due dates
  • Visualise the team's workload across multiple views: Kanban, Calendar, and List
  • Enforce resolution documentation before a ticket can be closed
  • Manage team members and control access through role-based permissions
  • Export ticket reports to CSV and Excel for management reporting
  • Track all activity, status changes, and comments per ticket

MethodEndpointAccessDescription
GET/AuthList users — Admins: full details; Agents: name/email only

PATCH/:id/disableAdminToggle user disabled status
DELETE/:idAdminDelete user with cascading data cleanup

AI Writing Assistance — /api/ai

MethodEndpointAccessDescription
POST/correctAuthImprove text using Gemma AI — returns corrected text preserving technical terms

Request / Response

POST /api/ai/correct
Body:    { "text": "need to chek the sever logs for erors" }
Returns: { "corrected": "Need to check the server logs for errors." }
The model used is gemma-3-12b-it via Google AI Studio (free tier: 14,400 requests/day). Requires GOOGLE_AI_API_KEY in backend/.env.

  • Full ticket editor: Title, Requestor, Description, Priority, Status, Assigned To, Due Date

7 Authentication & Authorization

How It Works

  • Session-based authentication using express-session
  • Passwords hashed with bcryptjs (12 salt rounds)
  • Sessions persist for 7 days
  • All API requests automatically send session cookies
  • Unauthenticated requests are redirected to the login page

User Roles

RoleCapabilities
ADMINSee all tickets, manage users, access Admin Panel, delete tickets and users
AGENTSee only assigned tickets, update ticket status, add comments, change own password

Security Rules

  • Disabled users receive a 403 error on login — cannot access the system
  • Agents cannot access the /admin route — redirected to Dashboard
  • Agents cannot call admin-only API endpoints — receive 403 response
  • Users cannot disable or delete their own account

8 Key Features

Ticket Management

  • Create tickets with Title, Requestor, Description, Priority, Assigned To, Due Date
  • System auto-generates Ticket ID and Creation Date
  • Resolution field mandatory before a ticket can be marked Done
  • Status workflow: To Do → In Progress → In Review → Done

Multiple Views

  • Dashboard — summary stats and recent activity at a glance
  • Kanban Board — drag-and-drop workflow management
  • Calendar View — timeline view for deadline management
  • List View — detailed table with search, filters, sorting, and export

Export

  • Export current filtered ticket list to CSV
  • Export current filtered ticket list to Excel (.xls)
  • Files automatically named with today's date

Team Management (Admin)

  • Create team members with a temporary password
  • Assign roles: Admin or Agent
  • Disable users — blocks login and hides from assignment, but preserves all data
  • Re-enable disabled users at any time
  • Delete users — all related tickets are safely reassigned

Collaboration

  • Post comments on any ticket
  • Automatic activity log records all status changes with timestamp and actor

AI Writing Assistance

  • Every description, resolution, comment, and due-date-reason field has a ✨ Improve button
  • Powered by Google AI Studio — Gemma 3 12B (free tier, no cost)
  • Fixes grammar, spelling, and sentence structure while preserving all technical terms
  • Suggestion is shown as a preview — users choose to Accept or Discard before any change is applied

Dashboard Analytics

  • Performance KPIs (all users): weekly tickets opened, weekly resolved, monthly tickets opened, monthly resolution rate %
  • Agent Performance chart (Admin only): grouped bar chart comparing tickets opened vs resolved per agent
  • Weekly Trend chart (Admin only): line chart showing last 4 weeks opened vs resolved
  • Monthly Trend chart (Admin only): bar chart showing last 6 months opened vs resolved

9 Business Logic

Ticket Visibility by Role

ADMIN → sees ALL tickets in the system AGENT → sees ONLY tickets assigned to them

Resolution Enforcement

  • The Resolution field only appears when Status is set to DONE
  • Resolution text is mandatory — save is blocked on both client and server if empty
  • Ensures every closed ticket has a documented resolution before it is closed

User Deletion Cascade

When an admin deletes a user, the following cleanup steps run in order:

  1. All tickets assigned to the user are unassigned
  2. All tickets created by the user are reassigned to the deleting admin
  3. All comments by the user are deleted
  4. All activity log entries by the user are deleted
  5. The user record is deleted

Assignment Dropdown Rules

  • Only active (non-disabled) users appear in the Assigned To dropdown
  • Applies to both Create Ticket and Edit Ticket forms
  • Tickets can be left unassigned

Overdue Detection

  • Calculated client-side: dueDate < today AND status !== DONE
  • Overdue tickets highlighted in red in List View
  • Warning icon shown on Kanban ticket cards

10 Docker & Deployment

System Architecture

Browser └── http://localhost (port 80) └── Nginx (frontend container) ├── / → serves React SPA (static files) └── /api/* → proxied to backend:3001 └── Express + Prisma (backend container) └── PostgreSQL 16 (db container)

Docker Compose Services

ServiceImagePortDescription
dbpostgres:16Internal onlyPostgreSQL database with persistent volume
backendCustom buildInternal (3001)Node.js + Express API server
frontendCustom build80:80Nginx serving the React app

Backend Dockerfile Startup Sequence

  1. Base image: node:18-slim (Debian — required for Prisma SSL compatibility)
  2. Installs openssl and postgresql-client
  3. Copies code and runs npm install
  4. Generates Prisma client
  5. On container start: waits for PostgreSQL to be ready
  6. Runs prisma migrate deploy to apply schema migrations
  7. Runs node seed.js to create default admin if not exists
  8. Starts node server.js

Commands

CommandWhen to Use
docker-compose up --buildFirst run or after any code changes
docker-compose upNormal start (no code changes)
docker-compose downStop all containers
docker-compose logs backendView backend logs for debugging

VariableDescription
